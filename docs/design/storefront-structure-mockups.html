Storefront · struktur og innhold · rev 9 — rule-compliant reference pass: extractions + edge states

Two pages, four decisions — rendered so you can point at what you want

Mocks use the design system verbatim: Geist, pure-white page, hue-250 neutrals, hairline dividers with borderless resting surfaces, shadow-soft reserved for the one focal booking surface. CTAs are the real Button default — monochrome near-black pill, 44px, font-medium — azure appears only on selected states, exactly like the app. Nothing in the mocks exceeds weight 500; hierarchy comes from size, spacing and the grey tiers. Interactive parts still work.

Question 1 · updated

Storefront overview — date-grouped agenda with filters

Row grammar copied from ClassPass's venue schedule: everything is a two-line stack — time over duration on the left, title over details·instructor in the middle, price alone on the right. No chips. A series says «8 økter» as plain second-line text; a drop-in class needs no label at all (like ClassPass, where booking a single class is the default). Scarcity is plain amber text under the price, not a pill.

Reference extraction — schedule page

Structural question: how do booking products structure a provider's public schedule — identity, filters, list anatomy — and what happens when a day or the whole page is empty?

Source screens: ClassPass venue schedule, Luma calendar, Time2book profile + empty schedule, Skillshare filtered-empty, SeatGeek «No games».

Layout skeleton: identity header → one control row → continuous date-grouped list. No sidebar, no day-strip; the date headers are the navigation.

Hierarchy: date headers dominate the scan; time + title primary per row; instructor/detail muted; price quiet at the right edge.

Component anatomy: row = time/duration stack · image · title/detail stack · price (+ scarcity text).

Spacing & density: airy rows, hairlines between rows, groups separated by a larger gap + hairline.

States observed: filtered-empty = one inline sentence + recovery link (Skillshare); truly empty = heading + one sentence with page structure retained (Time2book, SeatGeek); days without content simply don't render (Luma).

Decisions adopted: (1) date-grouped single list; (2) two-line stacks, no chips; (3) filters as the only control row; (4) empty states as plain text with a recovery action, never illustration cards; (5) empty days don't render.

Decisions rejected: ClassPass's day-tab strip (date groups already navigate); credits-in-a-square price (we show kroner); Luma's subscribe sidebar (no such feature).

Question 3 · redone

Course detail — three fresh architectures

Drawn from the second research pass: Luma's event page (booking is an inline panel in the reading flow, not a floating card), Posh (poster-led split canvas, essentials beside the artwork), and Spotify/Eventbrite (cinematic hero, display typography, price in a persistent bar instead of a sidebar). None of them use a right-rail price card. Tier reality check: exactly two sellable tiers exist today — «Hele kurset» (the course price) and the optional «Drop-in» toggle. The schema has headroom for passes and student pricing, but no seller UI creates such rows and checkout filters them out, so the mocks render only the two real ones. Component grammar is copied from shipped products, not invented: the summary ledger follows Fiverr/Typeform (muted labels left, hairline, modest total, black pill CTA), ticket rows follow Eventbrite (accent border marks selection, sublabel carries the constraint), and the fact rows follow Luma's two-line tiles.

Reference extraction — inline booking page (V1)

Structural question: where does booking sit in a single-column event page, how is the panel structured, and how do mature products render sold-out/closed without breaking it?

Source screens: Luma event page + Luma iOS, Eventbrite tickets + per-tier sold out, Fiverr summary, Posh closed state, SeatGeek no-tickets.

Layout skeleton: one centered column: art + essentials → registration panel inline → long-form content. No floating sidebar.

Hierarchy: title dominant; fact rows secondary; the panel is the page's only elevated surface; one price focal in the total row.

Component anatomy: panel = label header / option list / summary footer / full-width CTA / trust line.

States observed: whole event closed → CTA replaced by an inert message-pill, page otherwise unchanged (Posh); one tier sold out → the row stays, marked, others buyable (Eventbrite); nothing available → message plus a next-step action (SeatGeek).

Decisions adopted: (1) panel inline after essentials; (2) hairline ledger, not radio-cards; (3) summary footer with fee visible; (4) closed = CTA-replacement, never a hidden panel; (5) full tier keeps its row, marked «Fullt»; (6) sold out offers «se andre kurs» as the next step.

Decisions rejected: quantity steppers (one seat per booking here); promo-code field (no such feature); Luma's «You're in» post-state (belongs to checkout success, not this page).
